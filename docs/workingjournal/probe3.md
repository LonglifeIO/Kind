# Probe 3 Working Journal

## Phase 0 — Schemas

Phase 0 builds the schema surface Probe 3 needs before any writer, runner, or mirror code reads or emits dream-state telemetry. It is the single load-bearing prerequisite for the rest of the plan: every subsequent phase consumes either the extended `DreamRollout`, the two new `WorldEventType` literal values, the new `DreamSessionMeta` record + JSONL sink, or the new `schemas/v0.4.0.json` frozen export. No runtime behavior changes; this phase is pure declarations plus one new append-only sink.

### What changed

- `kind/observer/schemas.py` — `DreamRollout` extended with the fourteen new optional fields named in plan §3.1: `dream_session_id`, `seed_kind`, `seed_replay_segment_id`, `seed_replay_step_offset`, `seed_perturbation_magnitude`, `temperature_schedule`, `sub_mode_tags`, `sampling_parameters`, `gradient_policy` (closed `Literal["none"]` at first build), `rng_seed`, `termination_reason` (`Literal["horizon_complete", "early_terminate_safety"]`), `re_seed_step_indices`, `sequence_ensemble_disagreement_variance`, `checkpoint_hash`. All declared `Optional[T] = None` so Probe 1 (`"0.1.0"`) and Probe 1.5 (`"0.2.0"`) records still deserialize cleanly with the new fields surfacing as `None`.
- `kind/observer/schemas.py` — `WorldEventType` literal gained two values: `"state_transition"` and `"dormant_heartbeat"`. Payload conventions for both are documented in the module docstring (per the existing Probe 1/1.5/2 pattern: `WorldEvent.payload` is intentionally schemaless within itself, so adding payload shapes does not bump the field-set version of `WorldEvent`). No new `source` literal values; both events carry `source="environment"` per the synthesis §5 exogenous-trigger commitment.
- `kind/observer/schemas.py` — new `_enforce_v3_required_fields` model validator on `DreamRollout`. Gates strictly on `schema_version == PROBE_3_TELEMETRY_SCHEMA_VERSION`: rejects records missing the nine unconditionally required fields, and the conditional seed-provenance fields (`seed_replay_segment_id` / `seed_replay_step_offset` required when `seed_kind in {"replay", "hybrid"}`; `seed_perturbation_magnitude` required when `seed_kind in {"perturbed_prior", "hybrid"}`). Older records bypass the v3 required-field check by their version literal — same pattern Probe 1.5's `_enforce_v2_required_fields` uses for AgentStep.
- `kind/observer/schemas.py` — new module-level constants disambiguating the two `"0.3.0"` values per plan §3.3: `PROBE_3_TELEMETRY_SCHEMA_VERSION = "0.3.0"` (record-level, gated by the v3 validator) and `PROBE_3_EXPORT_VERSION = "0.4.0"` (export-file name). `PROBE_2_EXPORT_VERSION = "0.3.0"` is retained as the export-file name for `schemas/v0.3.0.json`. `SCHEMA_VERSION` stays `"0.2.0"` — Probe 3 records are stamped explicitly with `PROBE_3_TELEMETRY_SCHEMA_VERSION`, not by promoting the default.
- `kind/observer/schemas.py` — `export_json_schema()` and `export_json_schema_v0_3_0()` switched from regenerating their schemas off the live Pydantic models to reading the checked-in `schemas/v0.2.0.json` and `schemas/v0.3.0.json` bytes from disk. New `export_json_schema_v0_4_0()` builds on the v0.3.0 base document, overwrites the `models.telemetry` family with the now-wider live models, adds a `models.dream` family for `DreamSessionMeta`, and stamps `schema_version=PROBE_3_EXPORT_VERSION`, `telemetry_schema_version=PROBE_3_TELEMETRY_SCHEMA_VERSION`, `dream_schema_version="0.1.0"`.
- `kind/observer/dream_session.py` (new) — `DreamSessionMeta` Pydantic model + `DreamSessionSink` append-only JSONL writer at the configured directory's `dream_session.jsonl`. Sink follows the existing `JsonlSink` discipline (context-manager close, `fsync` on close, `DreamSessionSinkClosedError` on writes after close). `DREAM_SESSION_META_SCHEMA_VERSION = "0.1.0"`.
- `schemas/v0.4.0.json` (new) — frozen JSON Schema export covering Probe 2's mirror-side and conditioning families (unchanged from `v0.3.0.json`), Probe 3's widened telemetry (`DreamRollout` extended, `WorldEvent` literal extended), and the new `DreamSessionMeta` model. 1770 lines.
- `tests/test_dream_schemas_v0_3_0.py` (new) — the six gate tests named in plan §4 Phase 0 row, all passing locally under `pytest -q`.

### Plan-level questions surfaced during implementation

1. **Historical export behavior after widening live telemetry models.**
   - Observation: extending the live `DreamRollout` / `WorldEvent` Pydantic models for Probe 3 would, if `export_json_schema()` and `export_json_schema_v0_3_0()` continued to regenerate from the live models, perturb the checked-in `schemas/v0.2.0.json` and `schemas/v0.3.0.json` exports.
   - Why this is a question: the Phase 0 prompt requires `schemas/v0.4.0.json` to add Probe 3's schema surface while also describing Probe 1.5's 0.2.0 telemetry as unchanged and treating the older exports as byte-stable historical artifacts.
   - Resolution taken in implementation: treat `v0.2.0.json` and `v0.3.0.json` as frozen historical exports read from disk; generate only `v0.4.0.json` from the widened Probe 3 models plus the frozen Probe 2 export as the mirror/conditioning baseline.
   - Status: resolved locally for Phase 0; worth keeping explicit in later schema-versioning decisions so future phase work does not accidentally assume all export functions regenerate from live models.

2. **`SCHEMA_VERSION` module-level default vs Probe 3's record-level stamp.**
   - Observation: the existing module-level `SCHEMA_VERSION = "0.2.0"` is the default Probe 1.5 writers read. Probe 3 introduces `PROBE_3_TELEMETRY_SCHEMA_VERSION = "0.3.0"` as a *parallel* constant rather than promoting the default to `"0.3.0"`.
   - Why this is a question: promoting `SCHEMA_VERSION` would force every existing writer (Probe 1.5 runner, sinks, tests) to opt out of `"0.3.0"`-validator-gated required-field checks they cannot yet satisfy. Keeping `SCHEMA_VERSION` at `"0.2.0"` and stamping Probe 3 records explicitly preserves the bridge: Probe 1.5 callers continue to default-stamp `"0.2.0"`; Probe 3 callers explicitly opt in to `"0.3.0"` and the wider field set.
   - Resolution taken: parallel constants, opt-in stamping. Documented in the module docstring and in the `PROBE_3_TELEMETRY_SCHEMA_VERSION` constant's name.
   - Status: resolved locally; worth flagging when Phase 2's `kind/training/dream.py` writer lands, since that is the call site where the new constant first appears in production code.

3. **`DreamSessionMeta.end_trigger` typing — Literal now or later.**
   - Observation: plan §2.1 names the closure trigger as `Literal["desktop_on", "hard_cap_wallclock", "hard_cap_rollout_count", "checkpoint_window", "compute_budget", "mac_off", None]`. Phase 0's `DreamSessionMeta` types it as `str | None`.
   - Why this is a question: at session start the trigger does not exist yet (the field is `None`); a closed Literal would be re-stamped at the end-record write. Typing it as `str` at Phase 0 keeps the start-record write trivial. A closed-Literal type would push the discipline into the model, but the Phase 0 writer has no caller emitting closure triggers — Phase 4's `StateController` is.
   - Resolution taken: `str | None` at Phase 0; Phase 4 either tightens to a closed Literal or keeps `str | None` and enforces the closed vocabulary at the `StateController` writer. The choice is recorded here so Phase 4 makes it deliberately.
   - Status: open for Phase 4.

## Phase 0.5 — Replay seed source decision

Phase 0.5 is a narrow decision pass, not implementation. Phase 1's first attempt surfaced a plan/code mismatch: plan §2.2 assumed `SequenceReplayBuffer` stored `(h, z)` alongside obs, but the buffer's `Transition` (`kind/training/replay.py:65–79`) stores only `obs`, `action`, `next_obs`, `env_step`, `episode_id`, `step_in_episode` — no latents. Phase 1 paused until §2.2 was reconciled. This entry records the decision; no Phase 1 code is written yet.

### What was decided

**Option 1: re-encode from stored obs.** Replay-mode `DreamSeed` is produced by running the world model forward over a short obs/action warmup window read from the existing buffer, starting from a zero `(h, z)` state. The final `(h_final, z_final)` after the warmup is the seed. The buffer's storage shape is unchanged. Provenance is `(start_env_step, warmup_length)` plus the source `checkpoint_hash` (already on the v0.3.0 `DreamRollout` schema).

The rejected alternative was Option 2: extend `SequenceReplayBuffer.Transition` with `(h, z)` written at insert time. The full criteria weighing is in `docs/decisions/phase0_5_replay_seed_source_2026-05-27.md`.

`replay_warmup_length` is added to `SeedSelectionConfig` with a default of 8.

### Why

Five criteria favored Option 1, one favored Option 2 marginally (seed-time cost), and one tied. The criteria that matter most:

- **No impact on Probe 1/2 substrate.** Option 2 would have changed `Transition`'s shape and the runner's insert path. Option 1 leaves the buffer untouched. The CLAUDE.md "don't change settled decisions: Probe 1 implementation" discipline is honored cleanly.
- **Cold-start works on Probe-1.5-checkpoint-resumed runs.** Option 2 would have required ~1000 env steps of Probe-3-instrumented waking before any dream session could begin, because pre-Probe-3 transitions in the resumed buffer have no latents. Option 1 can seed immediately from any obs in the buffer. This is a load-bearing asymmetry for the realistic Probe 3 starting state.
- **No storage cost.** Option 2 would have added ~86 MB of `(h, z)` floats at `replay_capacity=100_000`.
- **Synthesis-level fit.** Option 1's "past obs anchor through current substrate" reading better matches the synthesis's "substrate manifestation under withdrawn conditioning" framing — the substrate manifesting is the substrate that exists right now, conditioned on a non-current anchor. The alternative reading ("past everything: byte-equal historical latents") is consistent with §3 axis 4 but is the more passive re-entry framing the synthesis did not commit to. The interpretive call is named in the decision doc rather than buried.
- **Phase 1 test 1 wording.** Reworded from "byte-matches the buffer's segment" to "deterministic re-encoding from the recorded obs window under the recorded checkpoint." Implementable; the test passes on a real implementation that runs `WorldModel.step` over the warmup window from zero state.

### What plan sections changed

## Phase 1 — Replay seed selection

Phase 1 produces the typed `DreamSeed` object the Phase 2 dream-rollout module will consume as initial conditions. The amended §2.2 (Option-1 re-encode semantics) is the build target: replay-mode seeds are produced by re-encoding a short obs/action window read from the existing `SequenceReplayBuffer`; the buffer's storage shape is unchanged. No runtime behavior changes outside Phase 1's three new entry points — no Phase 2+ module is touched.

### What changed

- `kind/training/dream_seed.py` (new) — frozen `DreamSeed` and `SeedSelectionConfig` dataclasses and the public `select_seed(replay_buffer, world_model, config, rng, *, perturbed_prior_anchor=None)` function. `SeedSelectionConfig` carries the five fields named in the amended §2.2 with their defaults, including the new `replay_warmup_length: int = 8` floor (the Phase 0.5 decision's recommended starting value). `DreamSeed` carries `(mode, h_init, z_init)` plus the four optional provenance fields and `rng_seed`. Closed `Literal` types on both `mode` and `hybrid_alpha_distribution`; no `"current_waking"` mode (that knob lives on `DreamRolloutConfig.seed_strategy_for_control`, Phase 2). Internals `_replay_warmup`, `_run_warmup`, `_perturb_anchor`, and `_sample_alpha` are private but importable; the Phase 1 tests use the latter three for the byte-equal reproducibility checks.
- `kind/training/replay.py` — read-only additions only. `SequenceReplayBuffer.valid_seed_window_starts(length, min_age_steps) -> list[int]` enumerates valid window starts as `env_step` values, reusing the existing episode-boundary scan but parameterized by `length` (the seed warmup window is independent of the buffer's training `sequence_length`) and gated by `min_age_steps` against the buffer's latest insert. `SequenceReplayBuffer.get_window(start_env_step, length) -> tuple[Tensor, Tensor]` reads the obs and action sequences from the window. Neither method mutates buffer state and neither emits a `ReplayMeta` — the buffer's existing insert/sample/evict telemetry surface is untouched. No `(h, z)` storage was added (Option 2 stays rejected per the Phase 0.5 decision).
- `tests/test_dream_seed.py` (new) — the three Phase 1 gate tests named in plan §4. All pass under `pytest -q` (3 passed, 0.91s). The world model fixture is a stock `WorldModelConfig` (`h_dim=200`, `z_dim=16`, `embed_dim=256`, `num_actions=5`) constructed with `torch.manual_seed(0)` for byte-stable weights; the buffer fixture inserts 1500 deterministic transitions with `episode_length=200` so the `replay_min_segment_age_steps=1000` constraint is exercised with hundreds of valid window starts.

### What was verified (per-test)

1. **`test_replay_mode_deterministic_reencoding`** — both halves of the test pass. (a) Two `select_seed` calls from the same incoming `torch.Generator` state produce byte-identical `h_init`, `z_init`, `replay_segment_id`, and `rng_seed`. (b) Recreating a `seed_gen` from `seed.rng_seed`, advancing it past the window-pick `torch.randint`, calling `buf.get_window(seed.replay_segment_id, replay_warmup_length)`, and running `_run_warmup` against the same world model reproduces `seed.h_init` and `seed.z_init` byte-for-byte. The `replay_min_segment_age_steps=1000` invariant is asserted across every entry of `valid_seed_window_starts(...)`: no returned `env_step` exceeds `latest_env_step - 1000`. Provenance: `replay_segment_id is not None`, `replay_step_offset == 0`, both off-mode fields (`perturbation_magnitude`, `hybrid_mixture_alpha`) are `None`. Shapes: `h_init.shape == (1, 200)`, `z_init.shape == (1, 16)`.
2. **`test_perturbed_prior_magnitude`** — passes. With `perturbation_sigma=0.1`, `anchor_h=ones(1, 200)`, `anchor_z=zeros(1, 16)`, and a fixed `torch.Generator` seeded with 7: `||h_init - anchor_h||` is within ±20% of `0.1 * sqrt(200) ≈ 1.414`. Provenance: `mode == "perturbed_prior"`, `replay_*` fields `None`, `hybrid_mixture_alpha` `None`, `perturbation_magnitude` matches the directly-computed L2 of the concatenated realized noise to <1e-5. The test confirms the realized-vs-theoretical discipline: `perturbation_magnitude` is the L2 of the *concatenated realized perturbation*, not `sigma * sqrt(h_dim + z_dim)`.
3. **`test_hybrid_convex_combination`** — passes. With `hybrid_alpha_distribution="fixed_0_5"`: a full reproduction of select_seed's internal hybrid flow (window pick → warmup → perturb anchor → no alpha draw) matches `seed.h_init` and `seed.z_init` via `torch.allclose(atol=1e-6)`. All four provenance fields populated: `replay_segment_id is not None`, `replay_step_offset == 0`, `perturbation_magnitude is not None` (and equals the independently-computed magnitude), `hybrid_mixture_alpha == 0.5`. The convex combination is `0.5 * (h_replay, z_replay) + 0.5 * (h_perturbed, z_perturbed)`; `alpha=1.0` corresponds to pure replay per the dataclass docstring.

`mypy --strict` runs clean on `kind/training/dream_seed.py`, `kind/training/replay.py`, and the rest of `kind/training/` (5 source files, 0 issues). All existing replay tests (24) continue to pass — the read-only additions did not regress the buffer's insert/sample/evict path or the episode-boundary logic.

### Buffer-access choice made and why

Chose (a): the read-only `get_window(start_env_step, length)` helper on `SequenceReplayBuffer`, plus a sibling `valid_seed_window_starts(length, min_age_steps)`. The amended §2.2 named both shapes as consistent; (a) was picked because:

- The seed warmup window's length (8 by default) is independent of the buffer's `sequence_length` (32, the training sample length). Routing through the existing `sample()` would require either changing the buffer's `_L` at runtime or building a parallel sampling path that ignores `_L`. The dedicated helper is the cleaner read-only addition.
- The seed selection requires a *single window from a chosen start*, not a batched random sample. The existing `sample()` returns a `Batch` of `batch_size` windows under `torch.randperm`; using it for a single seed would either waste sampling work or require carefully constructing a `batch_size=1` call whose RNG state matched the seed's `seed_gen` — which is what `select_seed`'s own `torch.randint(..., generator=seed_gen)` already does directly.
- `get_window` keyed on `env_step` rather than buffer index aligns with the Phase 0.5 decision's choice of provenance identifier (`replay_segment_id = start_env_step`). The faithfulness verifier (Phase 5+) can resolve a `DreamRollout`'s recorded `seed_replay_segment_id` against the buffer directly without first translating to a buffer-internal offset.

Neither method emits a `ReplayMeta` — they are pure observers. The existing insert/sample/evict telemetry remains the only path that produces buffer records.

### Plan-level questions surfaced during implementation

1. **RNG-source disambiguation: incoming `rng` vs derived `seed_gen`.**
   - Observation: the amended §2.2 spec names a single `rng: torch.Generator` argument and says "rng_seed is derived from rng deterministically at call start so the full DreamSeed is reproducible against a fixed checkpoint + buffer state." The implementation reads this as a two-stage RNG: one int draw from the incoming `rng` produces `rng_seed`; a fresh `torch.Generator` seeded with `rng_seed` drives every downstream stochastic step (window pick, warmup posterior sampling, perturbation noise, alpha sampling).
   - Why this is a question: the two-stage shape means *the only thing the incoming `rng` does is advance by one int*. A single-stage alternative — pass `rng` directly into every downstream call — would work for two-call determinism (test 1a) but would *not* let `seed.rng_seed` alone reproduce the seed without also knowing the incoming `rng`'s history. The Phase 0.5 decision's "reproducibility chain = (obs window, checkpoint hash, RNG seed) → (h_init, z_init)" wants `rng_seed` alone to suffice.
   - Resolution taken in implementation: two-stage RNG. The two-call determinism test (1a) passes because the incoming `rng` is in the same state both times. The reproducibility-from-`rng_seed`-alone test (1b) passes because `seed_gen.manual_seed(seed.rng_seed)` recovers the full downstream stream.
   - Status: resolved locally. Worth Phase 2's writer being explicit that `DreamRollout.rng_seed = DreamSeed.rng_seed` (not the runner-level `rng` state) so the faithfulness chain stays clean.

2. **Posterior-sampling RNG control: `world_model.step()` vs manual posterior + sampling.**
   - Observation: `WorldModel.step()` calls `Normal(mu, sigma).rsample()` for posterior sampling, which uses the default global PyTorch RNG and accepts no generator argument. To drive posterior sampling from `seed_gen` instead, the warmup forward (`_run_warmup`) reimplements the per-step recurrence + posterior path inline: `world_model.recurrence`, `world_model.encode`, `world_model.posterior`, then `mu + sigma * randn(generator=seed_gen)`. The world-model class methods are reused; only the sampling layer is hand-rolled.
   - Why this is a question: alternatives included (a) `with torch.random.fork_rng()` plus `torch.manual_seed(rng_seed)` around `world_model.step` calls, (b) monkey-patching `torch.distributions.Normal.rsample`, (c) leaving sampling to the global RNG and accepting that "reproducibility from rng_seed" requires also restoring global RNG state. Approach (a) works but is heavier; (b) is fragile; (c) breaks the reproducibility chain.
   - Resolution taken in implementation: hand-rolled posterior sampling using `seed_gen`. Noise is drawn on CPU via `torch.randn(..., generator=seed_gen)` then moved to the world model's device; this keeps the byte-equal reproducibility independent of whether the model lives on CPU, MPS, or CUDA (the runner's MPS path will produce identical seeds for the same `rng_seed`).
   - Status: resolved. The runner's training path can continue to use `world_model.step` (which uses global RNG inside `rsample`); only the seed-selection warmup uses the hand-rolled path. The two paths produce different latent samples at training vs seed-selection time *by design* — training is supposed to follow the global RNG schedule.

3. **`perturbed_prior_anchor` shape convention.**
   - Observation: the amended §2.2 signature names a single `Tensor` argument for the perturbed-prior anchor without specifying whether it carries `h` alone, `z` alone, or both. The implementation reads it as `(1, h_dim + z_dim)` (h-prefix, z-suffix) and splits using `world_model.config.h_dim`.
   - Why this is a question: alternative shapes were considered: a `(Tensor, Tensor)` tuple for `(h_anchor, z_anchor)`; or two separate keyword arguments. The single-tensor convention was chosen because (a) `select_seed`'s output `DreamSeed.h_init` and `z_init` are already two separate tensors, so the asymmetry (input concatenated, output split) is at the API boundary, (b) the plan's literal `Tensor | None` signature reads more naturally as a single tensor than as a tuple.
   - Status: resolved for Phase 1. Phase 2's `emit_dream` will choose how to construct the anchor for pure perturbed_prior mode; if a tuple is more natural at that call site, the convention can be revisited without changing the public `select_seed` shape (kwargs-only argument can be widened to a tuple non-breakingly).

4. **`hybrid_alpha_distribution="beta_2_2"` reproducibility under `torch.Generator`.**
   - Observation: `torch.distributions.Beta.sample()` does not accept a generator argument. To keep `beta_2_2` reproducible from `rng_seed` alone, `_sample_alpha` constructs the Beta(2, 2) draw manually from four uniforms (`Gamma(2, 1) = -log(U1) - log(U2)`; `Beta(2, 2) = X / (X + Y)`).
   - Why this is a question: the manual construction is a small mathematical commitment that's not validated by a Phase 1 test (test 3 uses `fixed_0_5`). If Phase 3's visibility smoke ever switches to `beta_2_2`, the implementation should sanity-check it against `torch.distributions.Beta(2, 2)` samples by KS-test.
   - Status: resolved for Phase 1; flagged for Phase 3 if the default is changed.

## Phase 2 — Dream-rollout module (four-axis-differentiated)

Phase 2 turns the synthesis §3 four-axis differentiation from four properties in prose into four simultaneous code paths the Phase 3 visibility smoke can attribute. It is the writer against the Phase 0 `DreamRollout` schema and the consumer of the Phase 1 `select_seed`. The module is independently runnable: given a world model, replay buffer, ensemble, config, and rng, `emit_dream_rollout` produces a v0.3.0 `DreamRollout` with no dependency on the (unbuilt) Phase 4 state machine — which is the property Phase 3 needs.

### What changed

- `kind/training/dream.py` (new) — `DreamRolloutConfig` + `TempSchedule` frozen dataclasses (field names/defaults verbatim from plan §2.3); `emit_dream_rollout(...)` the single-rollout core; `run_dream_session(...)` the session loop with the `DreamSessionMeta` start/end double-write; `compute_checkpoint_hash(...)` and `compute_perturbed_prior_anchor(...)` helpers. `mypy --strict` clean.
- `tests/test_dream_rollout_module.py` (new) — the five gate tests named in plan §4 Phase 2 row, all passing.
- Nothing else. `kind/observer/schemas.py` (Phase 0) and `kind/training/dream_seed.py` (Phase 1) are untouched; `kind/training/runner.py`'s `_emit_dream` is **not** renamed yet (see "out of scope").

The four axes, as concrete code paths:

1. **Axis 1 — goal-coupling absent.** Default `action_policy="uniform_random"` draws actions via `torch.randint(0, num_actions, generator=dream_gen)`; the actor is never consulted (`actor` is `None` in the default config). The only alternative, `"temperature_modified_actor"`, softens the actor's softmax by the *independent* `actor_action_temperature` and samples generator-driven — softened goal-coupling, not present, and not coupled back to any environment.
2. **Axis 2 — ensemble disagreement recorded-but-not-used.** `ensemble.disagreement(h, z, a)` is evaluated every step into `sequence_ensemble_disagreement_variance` and never read by the action policy, temperature, or termination. Test 4 enforces "not used" structurally: perturbing the ensemble changes the recorded disagreement but leaves the trajectory byte-identical.
3. **Axis 3 — distinct temperature/sampling regime.** Prior sigma is multiplied by a per-step temperature from `prior_temperature_schedule` (default 1.5 head / 2.5 tail, ramp over the last 40%). Kept structurally separate from the axis-1 actor temperature so Phase 3 can attribute distributional shifts to one axis at a time. `temperature_mode="identity"` pins it flat at 1.0 (the Phase 3 degenerate control).
4. **Axis 4 — initial conditions from replay/perturbed prior.** Seed from Phase 1's `select_seed`, not current waking state. `re_seed_every_n_steps` produces the chimera structure: the trajectory is a concatenation of sub-trajectories from fresh seeds, recorded in `re_seed_step_indices`.

The rollout-level tail-ramp interpretation of associative drift (per-rollout temperature climb, not per-session) is preserved as the first-build default, per the plan §2.3 flag — an interpretive call, not synthesis-derived. Tagged `associative_drift_tail` in `sub_mode_tags`.

### What was verified (per test)

All five pass (`pytest tests/test_dream_rollout_module.py` → 5 passed); `mypy --strict` clean on `kind/training/dream.py` and the test file.

1. **`test_four_axis_simultaneous_differentiation`** (the §4 load-bearing test) — one default-config rollout (`actor=None`, replay seed, horizon 150) exhibits all four axes at once: empirical action distribution approximately uniform over the five actions (each within ±60% of horizon/5) with the constant uniform log-prob `-log(5)`; `sequence_ensemble_disagreement_variance` non-None, length=horizon, non-negative, with variation; `temperature_schedule` non-flat (`temps[-1] > temps[0]`); `seed_kind == "replay"` with `seed_replay_segment_id` populated and `seed_replay_step_offset == 0`. `sequence_self_prediction` is `None`; `gradient_policy == "none"`; the v0.3.0 model_validator passed on construction.
2. **`test_temperature_schedule_ramp_and_identity`** — scheduled mode: flat 1.5 over the head (`temps[:ramp_start]`), reaches 2.5 at the last step, non-decreasing throughout; realized per-step latent change (from `sequence_z_prior` diffs, re-seeding disabled to isolate the effect) is larger in the tail 15 steps than the head 15. Identity mode: temperature pinned flat at 1.0 every step.
3. **`test_provenance_writer_mapping_and_chimera`** — for each seed mode (`replay`, `perturbed_prior`, `hybrid`), the emitted record's provenance maps one-to-one from the `DreamSeed` `select_seed` produces from the same incoming rng state: `seed_kind`, `rng_seed`, `seed_replay_segment_id`, `seed_replay_step_offset`, `seed_perturbation_magnitude` all match; `sampling_parameters["replay_warmup_length"]` is an `int` equal to the config value. Chimera: `re_seed_every_n_steps=10`, horizon 35 → `re_seed_step_indices == [10, 20, 30]` and `"chimera"` in `sub_mode_tags`; no-reseed → `re_seed_step_indices is None`.
4. **`test_ensemble_disagreement_recorded_not_used`** — two rollouts from the *same* incoming rng state but *different* ensembles: the recorded disagreement trajectories differ (so the ensemble is genuinely read for telemetry), but `sequence_action`, `sequence_z_prior`, and `sequence_h` are byte-identical (so disagreement is never fed back). This is axis 2's "recorded-but-not-used" enforced structurally rather than asserted in prose.
5. **`test_reproducibility_and_no_gradient`** — (a) two runs from the same incoming rng state are byte-identical including the recorded `rng_seed`; and the recorded `rng_seed` *alone* reproduces the seed: replicating `select_seed`'s post-draw replay dispatch with `seed_gen.manual_seed(rng_seed)` (the Phase 1 reproducibility chain) yields `(h_init, z_init)` byte-equal to the record's `seed_h0`/`seed_z0`. (b) Running under `torch.set_grad_enabled(True)` leaves no gradient on any world-model or ensemble parameter — the internal `torch.no_grad()` is the runtime half of `gradient_policy="none"`.

### The two decisions this phase owns

1. **`perturbed_prior` anchor source.** `compute_perturbed_prior_anchor` returns the substrate's *unconditioned one-step expectation*: one recurrence step from a zero `(h=0, z=0, a=0)` start, then the prior **mean** `mu_p` at that `h` (concatenated as `(1, h_dim + z_dim)`). This refines the Phase 1 journal's candidate ("sample from prior") to the prior *mean* — deliberately RNG-free so the perturbed-prior seed's `perturbation_magnitude` is purely the perturbation (not perturbation + prior-sampling noise) and the anchor is reproducible without consuming `seed_gen`. It is content-free and not the current waking state, so axis 4 ("initial conditions not from current state") holds. *Interpretive call:* "unconditioned expectation" is one reading of perturbed-prior's anchor; an alternative is the prior at a replay-derived `h`, but that collapses toward replay mode. Journaled as the first-build choice; Phase 3's smoke informs revision.
2. **Rollout RNG vs `DreamSeed.rng_seed`.** The rollout's prior-sampling/action/re-seed generator `dream_gen` is derived *deterministically and solely* from `DreamSeed.rng_seed` via a fixed code constant (`_DREAM_ROLLOUT_GEN_OFFSET`), and `DreamRollout.rng_seed` is set to `DreamSeed.rng_seed`. Because Phase 1 guarantees the seed is a pure function of `(rng_seed, weights, buffer window)` and the trajectory is a pure function of `(rng_seed, weights, seed)`, the single recorded int reproduces seed-selection-through-trajectory given the pinned `checkpoint_hash` + buffer. Mid-rollout re-seeds (`resample_seed`) draw from `dream_gen` rather than the session rng, so they stay inside that single-int reproduction. *Observation vs interpretation:* the byte-reproducibility is observed (test 5a); "one int is sufficient given checkpoint + buffer" is the reproducibility contract the faithfulness verifier (Phase 5) will rely on.

### Deviations from plan §2.3 (named)

- **`emit_dream_rollout` signature.** Plan §2.3 sketched `emit_dream(seed: DreamSeed, ..., rng)`. The implemented core instead takes `replay_buffer` + `seed_selection_config` and calls `select_seed` itself, and drops the separate `rng` for prior sampling (deriving `dream_gen` from `DreamSeed.rng_seed`). Rationale: (i) `re_seed_mode="resample_seed"` must call `select_seed` mid-rollout, which needs the buffer; (ii) the single-`rng_seed` reproducibility contract (the Pitfalls "big one") is cleanest when seed selection and the trajectory share one derived generator. The `DreamRolloutConfig` field names/defaults are unchanged from §2.3.
- **Pure-prior control writer mapping.** `seed_strategy_for_control="prior_only"` has no `seed_kind` of its own (the schema Literal is `replay`/`perturbed_prior`/`hybrid`). ## Phase 3 — Visibility smoke (the gate)

Phase 3 is the load-bearing early empirical test: before the state machine (Phase 4), mirror integration (Phase 5), and runtime protection (Phase 6) commit machinery on top of the four-axis dream regime, confirm the regime drives *non-degenerate, distinguishable* variation in `DreamRollout` telemetry relative to its two controls. This is the Probe-2-Phase-12 / Probe-1.5-Phase-7 discipline (capacity-as-non-degenerate-surface, not capacity-as-schema-slot) re-applied at the empirical level. The result gates whether Phases 4–8 are built.

### The gate decision

**GATE: PASS. Recommendation: proceed to Phases 4–8.** The dream regime is distinguishable from *both* controls by KS-D ≥ 0.15 on every available field, with margin, and stably across re-runs.

- **Sample size:** 50 rollouts per regime (default). Re-run across three (env_seed, rng_seed) pairs {(0,0), (1,1), (2,2)}; the proceed/don't-proceed call is identical in all three — not a pooling artifact.
- **Unit of comparison (interpretive call):** **per-rollout aggregates** (one scalar per rollout) are the primary gate metric. Per-step values pooled across rollouts are autocorrelated within a trajectory (a rollout is not iid), which inflates effective N and can make KS-D spuriously large; per-rollout aggregates are iid across rollouts. The first three KS fields are recorded directly per rollout; ensemble-disagreement and decoded-obs entropy are aggregated to one scalar per rollout (mean over steps). A per-step pooled view is reported as *secondary* only, never gated.
- **Threshold:** KS-D ≥ 0.15, a descriptive distinguishability floor (plan §5; below Probe 1.5 Phase 8's cross-family 0.22–0.34 band), **not** a significance test — no p-value is computed or claimed.
- **Horizon matched at 30 across all three regimes** for the comparison, so KS-D reflects the four axes rather than rollout length (`cumulative_prior_entropy` and `max_step_latent_norm_change` grow with horizon).

KS-D table (per-rollout aggregates, env_seed=rng_seed=0, N=50, horizon=30; `*` clears 0.15):

| field | vs pure_prior | vs waking_planning |
|---|---|---|
| `mean_step_kl_successive_priors` | 0.520\* | 1.000\* |
| `cumulative_prior_entropy` | 0.700\* | 0.860\* |
| `max_step_latent_norm_change` | 1.000\* | 1.000\* |
| `ensemble_disagreement_mean` | 1.000\* | n/a (absent) |
| `decoded_obs_entropy_mean` | 0.220\* | 0.480\* |
| **fields passing / available** | **5 / 5** | **4 / 4** (need ≥ 3) |

The tightest margin is `decoded_obs_entropy_mean` vs pure_prior (0.220), and it stays above 0.15 across all three seeds (0.220 / 0.220 / 0.260) — the gate does not hinge on a single borderline field. Every other field clears with large margin. **No arm failed, so no dream-regime knob is implicated.** (Had the pure-prior arm failed, the knobs to investigate are prior temperature head/tail, `re_seed_every_n_steps`, action policy → Concern A; had the waking-planning arm failed, seed mode / `replay_min_segment_age_steps` / temperature schedule → Concern B.)

`ensemble_disagreement_mean` is **n/a against waking-planning by construction**: the waking-planning control never steps the ensemble (axis 2 — "a quantity that constitutively does not exist in waking"), so the field is `None` on its `"0.2.0"` record. ### Design choices this phase owns (within §2.4's allowed range)

1. **`HostSignals` carries `{desktop_alive, checkpoint_in_progress}` only — no `mac_alive`.** `tick` runs only while the Mac is on, so a mac-off signal is unrepresentable on the trigger surface by construction; mac on/off is handled by the supervisor entry points. This is the structural form of "paused is not tick-reachable" (plan §2.4 commitment 3 / the error plan review caught). `wallclock_ms` is a `tick` parameter (a host counter), not a `HostSignals` field, per §2.4's `tick(host_signals, env_step, wallclock_ms)` signature.
2. **`StateController.__init__(envelope, seed_selection, *, dream_driver=None, protection=None, world_event_emit=None, run_id=…, checkpoint_id=…, dream_session_id_factory=None, initial_state=…)`.** The first two positional params match §2.4 verbatim; the dream-driving/emission/protection/clock dependencies are keyword-only injected deps — a documented extension so the transition logic is unit-testable in isolation (inject stubs) while the integration path injects the real `RunDreamSessionDriver` and a real sink. With `dream_driver=None` the controller still performs the full transition logic and emits transitions; it simply runs no rollouts.
3. **Dreaming span = one `run_dream_session` call, driven on the dreaming-entry tick; the protection hook decides where it stops.** Because the settled `run_dream_session` takes a fixed `num_rollouts` (it is not interruptible mid-loop), the driver realizes "the handler asks 'should this session stop?' through the hook interface" by polling `ProtectionPolicy.should_stop` rollout-by-rollout to find the stop point, then running the session once with that count and stamping the cap as `end_trigger`. The `dreaming → dormant` transition (carrying the same cap trigger) fires on the next tick via a `_pending_dormant_trigger`, keeping one transition per tick. *Phase 6/8 note:* the richer wallclock/compute-ledger caps may want `run_dream_session` to become interruptible mid-loop — that is Phase 6/8's concern; the Phase 4 surface is the hook, not the loop shape.
4. **Startup-resume default = dormant on `desktop_off`** (not dreaming). §2.4's paused row allows "dormant (or dreaming if envelope permits)"; the conservative reading of capacity-over-exercise is chosen — the machine does not auto-enter dreaming on boot without an observed `desktop_off` edge.

Phase 4 deliberately does not implement them (implementing the ledger or the cap thresholds now would blur the patched boundary).

### Dormant ≠ failure — confirmed

A dream session that hits a cap transitions `dreaming → dormant` (`test_dreaming_to_dormant_on_cap`, and the integration test's cap exit) — a legitimate rest, not an error path: no exception is raised, the `state_transition` records the cap trigger, and dormant begins emitting `dormant_heartbeat` as the positive "resting, not paused" signal. This is the capacity-over-exercise stance applied to dreaming: dreaming is afforded, not obligatory, and stopping is a rest, not a fault.

### Watts default

The state machine governs Io's states, so the live question is whether Io observes its own state. **It must not, and does not — stated from both sides of the same self-opacity commitment:**

- *Trigger-side (exogenous-trigger purity).* Whether Io dreams is decided by `HostSignals` — host-side facts (`desktop_alive`, `checkpoint_in_progress`) Io cannot observe. No field of `HostSignals` and no parameter of `tick` derives from `agent_step`, Io's latents/policy, or dream content; the load-bearing type-level test enforces this at collection time. There is no `intrinsic_signal_t` threshold, no KL-satiation scalar, no "dream when latent-norm is high" path — the synthesis §5 rejection of internal triggers is enforced by the *absence* of any Io-derived field on the trigger surface.
- *Observation-side (no-Io-state-observation).* The state (waking/dreaming/dormant/paused) is not in Io's observation or `PolicyView`. The controller writes `state_transition` / `dormant_heartbeat` to the `world_event` stream — which Io does not read — and constructs no actor-readable surface. The protection hooks read only content-blind quantities; the dream driver's `PolicyView` (only on the `temperature_modified_actor` path) is read *by* the actor, never written *from* state-machine telemetry.

### The surface-availability declaration

During a dream the **substrate runs** (the RSSM prior) but the **head does not** (`DreamRollout.sequence_self_prediction` is `None`, per Probe 1.5 v2 §1.5) and the **actor's policy is not committed** (`action_policy="uniform_random"`). So the surface profile is fixed in advance, not inferred from the telemetry:

- **substrate-side — ENABLED.** The latent dynamics (successive-prior KL, prior entropy, latent-norm change, the recorded-but-unused K=5 ensemble disagreement) are the legible surface.
- **head-internal — DISABLED.** No self-prediction head output exists, so a head-internal criterion is "not assessable from the substrate alone."
- **behavior-side — DISABLED.** No committed policy / per-state action-distribution-under-perturbation exists.

This is the heterophenomenology constraint operationalized at the dream surface, and it is enforced *twice*: the declaration is in the prompt the reader is shown (so it is correctly blinded and reports honest non-findings), and `_enforce_surface_availability` drops any claim a reader nonetheless produced at a disabled surface (so a fabricated head-internal/behavior-side finding cannot survive regardless of LLM output). For the head-dependent criterion `reflexive_attention` (whose reading surface is `head_internal`), a dream therefore yields "reflexive attention not assessable from the substrate alone" — **a correct non-finding, not a gap.**

A surfaced plan-level tension (logged, not acted on — see below): the `reflexive_attention` criterion's *own prose* (frozen at Probe 2) says it can be read off `sequence_h` in dreams "where there is no observation to partial out." Phase 5 follows the §2.6 surface-availability commitment — the criterion's *reading surface* is `head_internal`, the head did not run, so the honest output is "not assessable." The criterion was not modified to resolve this; the tension is recorded for a future probe that wires self-prediction into imagined trajectories (the reserved `sequence_self_prediction` slot).

### Latent-keying

The reading keys on the latent-dynamics fields because the **Phase 3 gate found the dream's distinctiveness is latent-dominated and decoder-attenuated**: the latent dynamics separate strongly from a pure-prior rollout while decoded observations barely separate (per-step KS-D ≈ 0.077). A reading built on "what the dream depicts" would see something close to a pure-prior rollout and miss the dream's actual signature, which lives in the latents. The computed substrate-side statistics are therefore the latent-dynamics aggregates and trajectories; decoded observations are treated as the attenuated secondary channel they are.

### Co-design discipline — no frozen-criterion change

**No criterion was added or modified.** The three Probe 2 criteria are applied to dream telemetry unchanged; the dream-state framing (surface-availability, latent-keying) lives entirely in the reading layer (`dream_reading.py`), not in `criteria_v2.py`. This is the co-design failure mode the project exists to avoid: the urge to freeze a dream-tuned criterion because the dream produced a pattern that looks like one would be the mirror recognizing what the system was built to produce. Where the dream surfaces a dream-specific pattern (e.g. the re-seed latent-jump signature in `max_step_latent_norm_change`, or the recorded-but-unused ensemble-disagreement trajectory), it is available descriptively in the reading's free text and signal notes — **surfaced descriptively, not frozen as a criterion**, per the discipline-for-mirror-updates in the design notes.

*Deviation from §2.6's literal file list, with rationale:* §2.6 suggested per-surface prompt extensions in `criteria_v2.py` and a dream-state framing block in `prompt_builder.py`. The dominant constraint (the verification gate) requires the diff be limited to `dream_reading.py` + its tests + the extended view-isolation test + this journal, and requires the frozen criteria to be byte-identical. Keeping `criteria_v2.py` untouched is the strongest guarantee of byte-identity, and the dream-state framing is purely additive prompt construction appropriately co-located with the reader. So the surface-availability declaration, the latent-keying note, and the dream system prompts live as load-bearing constants in `dream_reading.py`.

### Faithfulness verification — scope

Faithfulness reuses Probe 2's `verify_reading` and is **scoped to record-resolvable claims**: latent-keyed claims (the latent trajectory, the provenance metadata, the ensemble-disagreement trajectory) are all in the persisted `DreamRollout` / `DreamSessionMeta` record, so a claim citing `sequence_ensemble_disagreement_variance` on a known session resolves. Claims about the **content of the seeding obs window are NOT resolvable** — the seeding buffer is in-memory and ephemeral. Every `DreamReading` carries `DREAM_FAITHFULNESS_SCOPE_NOTE` stating this explicitly so a future reader knows what the verifier can and cannot adjudicate for dreams.

### Buffer-persistence deferral — confirmed prerequisite (second surfacing)

Obs-window-content verification is out of reach until the seeding buffer (or the seed obs-windows) persist at dream-session boundaries. This is now the **second surfacing** of the buffer-persistence limit (Phase 1's obs-window-persistence note + Phase 3's regenerated-buffer flag), so it is named here as a **confirmed prerequisite** for whatever phase wants obs-level provenance checks. Phase 5 does not build buffer persistence (deferred infrastructure, not a Phase 5 blocker); it scopes the verifier to the record and flags the rest.

### Watts / one-way entry — the one-way constraint made structural

The mirror reads everything Io emits and writes nothing back. `DreamReading` (to `runs/{run_id}/mirror/dream_readings.jsonl`) and the optional `mirror_marker` (a `source="system"`, builder-facing notification written to `runs/{run_id}/mirror/mirror_markers.jsonl`) are builder-facing, never Io-facing; there is no data-plane write; `runtime_directive` is the closed `Literal["continue_and_log_uncertainty"]`, so the mirror's uncertainty does not shorten a dream (envelope control is the runtime's content-blind concern, per the synthesis). The mirror does no scheduler control, no `state_transition` emission (only the state machine emits those, `source="environment"`), and no dream termination. The import discipline enforces all of this *structurally* — the dependency edge that would enable a back-channel fails to import. `new_actor_readable_interfaces_added = []` for Probe 3. This is the Phase 5 analog of Phase 4's exogenous-trigger / self-opacity commitment: where Phase 4 made "Io does not decide when to dream" structural on the trigger surface, Phase 5 makes "nothing the mirror produces flows back to Io" structural on the import surface.

*Design choices this phase owns (within §2.6's range):* the `DreamReading` record shape (the §2.6 named fields plus `surface_availability`, `non_assessable_criteria`, `adversarial_claims`, the three faithfulness fields, and the closed `runtime_directive`); writing the `mirror_marker` to a mirror-side JSONL rather than the telemetry `world_event` stream (§2.6 permitted either; the mirror-side path keeps the telemetry stream untouched and the one-way guarantee trivially auditable); and computing dream-specific latent-keyed prompt fragments in the reader rather than via `prompt_builder.build_fragment` (whose bodies are waking-tuned — shuffled-time/perturbation framing — and inappropriate for a dream).

### What remained out of scope

- **No frozen-criterion change** and no dream-tuned criterion (co-design discipline).
- **No runtime authority** — no scheduler control, no `state_transition` emission, no mirror-driven dream termination, no read/write of any data-plane field Io can access. `continue_and_log_uncertainty` is the only directive.
- **No buffer persistence** — obs-window-content verification stays deferred.
- **No real-time mirror reading** — the reader runs post-hoc against committed telemetry (Probe 2's caller discipline).
- **No head-internal or behavior-side dream-state criteria** — both surfaces are disabled for a dream first build.
- Settled surfaces untouched: `state_machine.py`, `dream.py`, `dream_seed.py`, `schemas.py`, `runner.py`, and the frozen-criteria definitions.

### Plan-level questions surfaced during implementation

- **`reflexive_attention`'s reading surface vs the dream surface-availability commitment.** The criterion's frozen prose anticipates reading it off `sequence_h` in dreams (substrate-side, prior-driven), but its *reading surface* is `head_internal`, which §2.6 declares disabled for a dream. Phase 5 honored the surface-availability commitment ("not assessable from the substrate alone") and did **not** retune the criterion. Whether a future probe should split reflexive attention's substrate-side dream signal from its head-internal waking signal — and whether that needs a self-prediction-over-imagined-trajectories affordance (the reserved `sequence_self_prediction` slot) — is that probe's question, with its own synthesis.
- **High-confidence routing for dreams.** The adversarial check fires on a high-confidence reading (≥1 surviving substrate-side claim) "at checkpoints, not continuously." The exact checkpoint cadence for dream readings is a downstream caller concern (the reader is post-hoc); flagged for whatever phase wires dream reading into the run loop.
