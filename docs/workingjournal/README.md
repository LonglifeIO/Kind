# Kind — Working Journal

Hand-written notes captured during each probe's build and during the runs
that follow. The journal is the project's working memory: the record of
what was decided when, what surprised, what's now closed, and what's now
newly open.

The journal is the bridge between probes. Each new probe's research
prompt is sharpened by reading what surfaced in the prior probes; the
synthesis after a probe's research stands on the journal entries that
preceded it. Without this layer, the lessons of each probe stay implicit
and don't propagate forward.

The journal is not generated by Claude or any other tool. It is the
builder's own record. The tools (``kind/observer/eyeball.py``) are for
reading what the runner wrote; the journal is for writing down what the
builder learned.

## Structure

One file per probe, plus pre-probe risk-framing files where relevant.
Cross-probe notes (e.g., a question surfaced during one probe that bears
on a later one) live in the file where they were surfaced, with a clear
cross-reference. Future-probe research starts by reading all prior probe
journals.

Existing files in this directory:

- ``guide.md`` — the original framing of the journal's role and
  rhythm. Short; read once, then write.
- ``pre-probe1.md`` — risk framing before Probe 1's build. Failure
  modes named going in; structural choices recorded mid-build that the
  smoke will end up testing.
- ``probe1.md`` — Probe 1's build journal (one section per phase) and
  its post-smoke entries.
- ``pre-probe4.md`` — early surfacing of the builder-as-perturbation
  mechanism question, captured during Phase 3c when it came up. Carries
  forward to Probe 4 research.

Convention for future probes:

- ``pre-probe2.md`` — risk framing before Probe 2's build, written
  alongside Probe 2's research synthesis.
- ``probe2.md`` — Probe 2's build journal.
- ``pre-probe3.md`` / ``probe3.md`` and ``pre-probe4.md`` / ``probe4.md``
  in the same shape.

The pre-probe file is where doubts and predictions go before code lands.
The probe file is the running log of what was actually built and what
surfaced. The two are paired: when the smoke runs, the pre-probe file is
what tells you whether the surprise is a failure mode you'd named or one
you hadn't.

## Entry conventions

Each phase's entry covers:

- **What was built.** Briefly. The plan is the spec; the entry doesn't
  re-state it.
- **Structural decisions worth recording.** The ones the smoke or future
  probes will end up testing. Not every choice — the journal is not the
  diff. The ones that have *judgment* in them: where the spec was
  permissive, where two options were both defensible, where a default
  was picked under uncertainty.
- **What's now closed.** Decisions made during the phase that are now
  settled at the project's current scale.
- **What's now newly open.** Questions surfaced during the phase that
  carry forward — into the next phase, the next probe, or the running
  list of "watch for this in the smoke."

Post-build entries (after running the smoke and reading what surfaced)
cover:

- **What ran end-to-end and what surprised.** Distinguish what the
  synthesis predicted from what the build revealed.
- **Specific defaults that the smoke promoted from "default" to
  "settled."** Per the implementation plan §6 framing.
- **What's now newly open that the synthesis didn't anticipate.** These
  become the next probe's starting context.
- **A note on whether the substrate decision held up operationally.**
  The architectural question stays alive across probes; each probe is
  evidence about it.

Entries are dated. Author is the builder. Two- to four-bullet entries
are fine for small phases; longer ones for phases where the judgment
calls were heavier. The discipline against confirming prior beliefs
(``Kind_design_notes.md``, "Observer layer (me)") applies here: when
something surprises, stay with the surprise rather than immediately
folding it back into what was already expected.

## Reading the runner's output

The journal is hand-written, but the data that informs it comes from the
runner's telemetry. The Phase 7 eyeball helpers under
``kind/observer/eyeball.py`` are the way to read it:

```
python -m kind.observer.eyeball summary <telemetry_dir>
python -m kind.observer.eyeball recent  <telemetry_dir> [-n N]
python -m kind.observer.eyeball events  <telemetry_dir>
python -m kind.observer.eyeball episode <telemetry_dir> [-e EPISODE_ID]
python -m kind.observer.eyeball dream   <telemetry_dir> [-i INDEX]
```

These print plain text that copy-pastes cleanly into a journal entry.
There are no dashboards, no plots, and no rich formatting at Probe 1
(per the plan §2.11 "no visualisation beyond eyeballing JSONL/Parquet").
ASCII rendering of decoded dream observations is the single visual aid;
everything else is scalar summaries.

The mirror caller (``kind/mirror/caller.py``) is the LLM-side reader. It
shares the digest function with the eyeball helpers
(``kind/observer/digest.py``) — which means what the mirror sees and
what the human sees are the same shape. Reading what the mirror is about
to read is a one-line invocation; the mirror is one consumer among
several, not a privileged one.
